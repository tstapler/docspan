# Architecture Review: wedding-planning-workflow
**Date**: 2026-07-18
**Verdict**: CONCERNS

## Blockers

None. The prior BLOCKER is resolved.

**Verification of the repair (prior Blocker: comment-risk gate split across CLI/backend fetches)**:

- **Single-fetch invariant, verified structurally, not just asserted**: Story 1.2.3 (`plan.md` lines 305-357) introduces `GoogleDocsBackend._build_push_plan()`, which performs exactly one `get_document()` and one `list_comments()` call and derives `requests`, `entries`, and `high_risk` from that same fetch (Task 1.2.3a/1.2.3c). `push()` calls `_build_push_plan()` once, blocks on `plan.high_risk and not force` *before* touching `batch_update()`, and otherwise writes using `plan.requests`/`plan.doc["revisionId"]` from that identical snapshot. Task 1.2.3d's acceptance criteria go further than a design description — they specify a regression test that asserts `get_document.call_count == 1` and `list_comments.call_count == 1` on the blocked path, which is exactly the right test to pin this invariant so a future refactor can't silently reintroduce a second fetch. `preview_push()` also calls `_build_push_plan()`, but independently (its own fetch) — it is never passed into, shared with, or consulted by `push()` (Story 1.2.3's framing paragraph is explicit that this "share a fetch across CLI and backend" shape is exactly what caused the original bug, so it's deliberately not done). CLI's `--dry-run` path (Task 1.2.4a) and the real-push path (Task 1.2.4b) are cleanly separated: the CLI no longer calls `preview_push()` before deciding whether to call `orchestrate_push()`.
- **Residual TOCTOU window — honestly documented, not silently ignored**: A smaller race remains (a comment landing in the millisecond gap between `_build_push_plan()`'s fetch and `batch_update()` actually committing, since Drive comment metadata isn't covered by `writeControl`/`revisionId`). This is disclosed in three places, not one: ADR-002's Consequences section (explicitly labeled "residual, smaller TOCTOU gap remains and is accepted, not hidden"), the Risk Control section's `CommentCountBackstop` bullet (plan.md line 87), and `feature-gap-report.md` item 8 (Story 3.2.1, plan.md line 520), which spells out that the backstop is a *detector*, not a *preventer*. This is the honest disclosure the task asked for — the repair doesn't claim to have eliminated TOCTOU, only to have shrunk it from "an entire CLI invocation" to "milliseconds inside one write," and says so explicitly.
- **`force` vs `RevisionGuard` — correctly kept independent**: Story 1.2.4's acceptance criteria state directly that "`--force` only bypasses the *risk-flag* block, not the `RevisionGuard`" (plan.md line 370), and Task 1.2.3c requires the stale-revision `try`/`except` from Story 1.1.2 to be preserved intact when `push()` is rewritten. `batch_update()` is always called with `required_revision_id=plan.doc["revisionId"]` regardless of `force`. These are two independent guard clauses (comment-risk vs. optimistic-concurrency) and the repair does not conflate them.

## Concerns

- [ ] **Comment data is still `list[dict]` / raw Drive JSON — prior Concern not addressed, not declined.** The repair fixed the *duplicate-diff-computation* concern (below) but left this one exactly as before: `GoogleDocsClient.list_comments(doc_id) -> list[dict]` (plan.md line 280), `PushPlan.comments: list[dict]` (line 327), and `find_high_risk_paragraphs(entries, comments)` still reaches into `comment["quotedFileContent"]["value"]`/`comment["author"]["displayName"]`-shaped dict access (line 296) rather than a parsed `Comment` value object. Unlike the mistune-drift concern below, there is no note anywhere in `plan.md` acknowledging this tradeoff or explaining why it was left as-is — it reads as simply unaddressed rather than a reasoned decline. Not a blocker (the substring-match logic doesn't get meaningfully safer with a typed wrapper), but worth a one-line acknowledgment or the small fix.
  **Recommendation**: Either add the small `Comment` dataclass at the `list_comments()` boundary as originally suggested, or add a sentence to the Domain Glossary's `CommentSnapshot` row explicitly declining to type it (e.g. "kept as raw dict — low payoff for a single, already-tested consumer function") so it reads as a decision rather than an oversight.

- [ ] **`research/build-vs-buy.md` §1b still says "Recommended" for the mistune `task_lists` plugin — prior Concern not addressed.** ADR-001 and `plan.md` still correctly avoid enabling `task_lists` (unchanged, still sound), but `build-vs-buy.md` itself was not touched: §1b's verdict (line 70: "**Recommended.** Adopt `mistune`'s `task_lists` plugin...") and the summary table (line 225: "mistune `task_lists` plugin (already a dependency) | **Recommended**") still point the wrong way, with no "superseded by ADR-001" note. This is the same gap flagged in the prior review, carried forward unchanged.
  **Recommendation**: Same as before — add one line to `build-vs-buy.md` §1b/its summary table noting it's superseded by ADR-001's literal-text decision.

- [x] **Duplicate diff computation between `diff_summary()` and `find_high_risk_paragraphs()` — resolved.** `find_high_risk_paragraphs` now takes `entries: List[DiffEntry]` (the output of `diff_summary()`) instead of re-running `difflib.SequenceMatcher`/`_text_key` (Story 1.2.2, Task 1.2.2b, plan.md lines 282, 296). The Domain Glossary's `CommentCrossReference` and `HighRiskParagraph` rows both call this out explicitly as the fix. Matches the recommendation from the prior review.

- [x] **`hasattr` duck-typing for `preview_push()` vs. `Protocol` — reasonably downgraded, not literally fixed, but the underlying risk is gone.** The plan did not switch to `typing.Protocol`/`isinstance` as recommended; `cli/main.py` still uses `hasattr(backend, "preview_push")` (Task 1.2.4a). However, the Pattern Decisions table (plan.md line 68) explicitly addresses why this is now acceptable: because `preview_push()` is purely cosmetic and no longer gates whether a real push proceeds, a typo'd/missing method now degrades to "no dry-run rendering, falls back to the stub message" rather than "safety gate silently bypassed" — which was the actual substance of the original concern. This is a reasoned, explicit decline rather than a silent drop, and the severity of the original concern is structurally lower now that Story 1.2.3 moved enforcement into `push()` itself. Acceptable as-is; `Protocol` would still be a nice-to-have but is no longer safety-motivated.

## Nitpicks

- `DiffEntry.style` / `HighRiskParagraph` fields still typed as raw `str` (primitive obsession) — unchanged from prior review, consistent with existing `DocsParagraphNode.style: str` convention, not a new issue.
- Task 1.2.2c's test-coverage note is now a firm commitment ("this is a firm commitment, not conditional," plan.md line 300) — prior nitpick resolved.
- The fail-closed condition (`if plan.high_risk and not force`) in Task 1.2.3c is still a bare inline boolean guard rather than a named predicate (e.g. `PushPlan.is_blocked(force)`). Still fine at today's single call site; same low-priority suggestion as before if a second caller ever needs the identical check.
