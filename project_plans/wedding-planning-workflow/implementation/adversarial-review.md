# Adversarial Review: wedding-planning-workflow
**Date**: 2026-07-18
**Verdict**: CONCERNS

The prior BLOCKER (checklist-shape verification from a single sampled line) is genuinely resolved by the repair pass. No new blocker-level issue was found. One new CONCERN emerged directly from re-checking how the fix interacts with the Blocker-1 (architecture-review.md) `PushPlan`/`push()` redesign: the mixed-doc fallback's "flagged high-risk" language for native-glyph paragraphs never actually reaches the code path that enforces `high_risk` at push time — it's a pull-time-only disclosure, not a push-time gate, and that distinction isn't stated plainly anywhere in the plan or ADR.

## Concerns

- [ ] **The mixed-doc fallback's "flagged high-risk" language for native-glyph paragraphs is disclosure-only — it never reaches `push()`'s automated `high_risk` gate, so the exact corruption scenario the original BLOCKER worried about (a literal `[x]`/`[ ]` marker layered onto an already-present, unrelated checkbox glyph) is still only prevented by Tyler's memory, not code.** Trace the two mechanisms: `HighRiskParagraph`/`PushPlan.high_risk` (Story 1.2.2, `find_high_risk_paragraphs(entries, comments)`) is defined purely in terms of comment overlap — it filters `DiffEntry`s for a `quotedFileContent.value` substring match and has no parameter, field, or code path that consults `ChecklistParagraphSurvey`'s findings at all. `MixedChecklistWarning` (Task 0.1.2b), by contrast, is explicitly scoped to **pull only** — "A pull-time `WARN` log line... emitted for any paragraph `ChecklistParagraphSurvey` found to be a native `BULLET_CHECKBOX` glyph" (Domain Glossary, `plan.md:53`), implemented in `docs_structure_parser.py`/`converter.py`, not `backend.py`/`push_preview.py`. `_build_push_plan()` (Task 1.2.3a) never re-derives or consults glyph shape — it only calls `diff_summary()` and `find_high_risk_paragraphs()`. So if Tyler (or Claude drafting a digest on his behalf) reads the pulled markdown's `<!-- docspan: native checkbox glyph, state not readable -->` marker, doesn't fully register its implication, and edits that exact paragraph — e.g. rewording a schedule line that happens to sit on a native-glyph bullet, or literally prepending `[x]` to "check it off" the way every other checklist line in the doc is checked off — `push()`'s gate sees an ordinary `"change"` `DiffEntry` with no comment overlap and proceeds without warning or block, `--force` never even enters the picture. ADR-001 (`plan.md`/ADR-001 Decision section) calls this "treated as high-risk / requires manual handling," which reads as if it plugs into the same safety machinery `HighRiskParagraph` uses elsewhere in this plan — it doesn't. Contrast this with the "isolate checklist-only pushes" gap, which the plan explicitly and plainly labels "documented operator discipline, not tool-enforced" in the Risk Control section — no equivalently plain statement exists for this native-glyph case; the runbook's Task 3.1.1b Do/Don't rules don't mention "never edit or push a paragraph carrying the native-checkbox marker comment" either. — **Recommendation**: pick one of two cheap fixes. (a) Minimal: add one explicit sentence to ADR-001's Decision section and one Do/Don't rule to the runbook (Task 3.1.1b) stating plainly that `MixedChecklistWarning` is pull-time-only disclosure, is **not** consulted by `push()`'s `high_risk` gate, and that Tyler must never edit a marker-commented paragraph via docspan (Docs UI only) — closing the honesty gap even without new code. (b) Slightly more robust: extend `find_high_risk_paragraphs()` (or add a sibling check inside `_build_push_plan()`) to also flag any changed `DiffEntry` whose `current_text` lacks a literal `[ ]`/`[x]` prefix but whose paragraph was identified by a (re-run, live) glyph check as checkbox-shaped, folding it into the same `high_risk` list so `--force` is required there too — this is the option that actually closes the corruption scenario at the same enforcement point as everything else in Epic 1.2, rather than relying on a second, disconnected safety story.

- [ ] **requirements.md's "keep the local markdown file under git" Risk Control item still has no corresponding task in plan.md.** Carried forward unchanged from the prior review — a repo-wide `grep -n "git add\|git commit\|git init\|under git\|git repo"` against the updated `plan.md` returns no matches. This wasn't one of the five items the repair pass claimed to address, and it appears it was simply not touched. — **Recommendation**: still just one line in the runbook's CLI sequence (`git add wedding.md && git commit` after each `docspan pull`) plus confirming/creating a git repo for the local markdown, most naturally as a Phase 0 task alongside Task 0.1.1a.
