## Test Stack
- **Unit**: `pytest` (already the repo's framework â€” see `pyproject.toml`'s `[tool.pytest.ini_options]` and `tests/*.py`), plain `test_*` functions grouped into `Test*` classes for multi-scenario areas (matches existing `tests/test_cli.py`/`tests/test_orchestrator.py` convention), `unittest.mock.MagicMock`/`patch` for doubles (matches `tests/test_cli.py`'s existing pattern).
- **Integration**: `pytest` + `unittest.mock` for the Drive/Docs service layer where mockable (`list_comments`, `batch_update` call shape); for the genuinely-live-only pieces (Phase 0 `ChecklistParagraphSurvey`, Story 2.2.1's round-trip and comment-collision cycles, CommentCountBackstop's real-API behavior) there is no automated integration test â€” these are manual runbook steps against the `ScratchDoc`, tracked below, because they depend on live Google Docs/Drive API behavior (undocumented glyph resolution, real comment anchor persistence) that cannot be faithfully simulated with mocks without begging the question the verification exists to answer.
- **E2E / UX**: N/A â€” personal CLI tool, manual verification against a scratch Google Doc instead (see Manual Verification Steps section).
