# Research: Pitfalls — wedding-planning-workflow

**Agent**: Research Agent 4 (Pitfalls)
**Date**: 2026-07-18
**Scope**: Checklist round-trip + comment-anchor preservation in `src/docspan/backends/google_docs/*.py`; personal pull→summarize→edit→pull→dry-run→push workflow against one live, multi-editor wedding-planning Google Doc, ~11 days out.

## 0. Top finding — the planned safety net (`--dry-run`) doesn't exist yet

`requirements.md`'s Risk Control leans entirely on: *"review `docspan push --dry-run` output before the real push... if a push looks destructive, stop."*

`src/docspan/cli/main.py:107-111` shows `push --dry-run` is currently a **no-op stub**:

```python
if dry_run:
    console.print(
        f"[yellow]dry-run[/yellow]  {mapping.local} → [{mapping.backend}] {mapping.remote_id}"
    )
    continue
```

It prints one line and `continue`s — it never calls `backend.push()`, never fetches the live doc, never runs `DocsRequestBuilder().build(...)`, and shows zero diff. There is also no separate `docspan diff` command anywhere in `main.py` (confirmed: full command list is `push`, `pull`, `status`, `auth-setup`, `conflicts list`, `conflicts resolve` — no diff/preview command exists).

**Implication**: the entire "always review the dry-run diff before pushing to a live, actively-commented-on doc" mitigation Tyler is counting on does not currently exist. This is not a nice-to-have gap — it is the single control standing between an automated push and silently clobbering Nora's/Bekah's edits or comments. This must either be implemented for real (build the request list against a freshly-pulled doc and print a human-readable summary of deletes/inserts/style-changes, without calling `batch_update`) or the workflow must be redefined to use something else (e.g., `git diff` on the local markdown file between the two pulls, which is a much weaker signal since it doesn't show what the *push* will actually do to the live doc — see §2).

## 1. Google Docs API batchUpdate pitfalls

**Index-shift handling — already correct, but fragile to extend carelessly.**
`docs_request_builder.py` computes every request's `startIndex`/`endIndex` from the **pre-mutation** document (the `current_nodes` list from a single `get_document()` call), then sorts the full `all_requests` list descending by `startIndex` before sending (`build()`, lines 82-87). Google's batchUpdate applies requests **in array order, sequentially, with each request's indices interpreted against the document state after all prior requests in the batch have applied** (confirmed via Docs API docs: requests process in the order given, and later index-based requests must account for earlier requests' effects within the same batch). Applying strictly highest-original-index-first is the standard correct mitigation, and this codebase already does it correctly — including a subtle-but-correct trick in `_make_insert_requests`: multiple new nodes inserted at the *same* `insert_at_index` are iterated in `reversed()` order, then rely on Python's stable sort (ties preserve original append order) to land in the right final left-to-right order.

**Sharp edge for the checklist feature specifically**: this correctness depends on the invariant "every request's indices reference the single, unmutated document read via one `get_document()` call." Any new checklist-toggle logic that:
- reads indices from a *stale* cached parse (e.g., reusing indices from the earlier `docspan pull` instead of the fresh `get_document()` call inside `push()`), or
- emits `createParagraphBullets`/`deleteParagraphBullets` ranges computed independently from the paragraph's `insertText`/`deleteContentRange` ranges (e.g., if a toggle is implemented as an in-place partial edit rather than routing through the existing delete+insert "replace" path),

will silently break the ordering invariant and reintroduce the classic index-shift bug. Keep any new checklist requests inside the same `all_requests` list that gets the global descending sort — do not special-case them into a second batch or a second `batch_update()` call.

**`BULLET_CHECKBOX` exists, but has a real read-back gap.** The Docs API `createParagraphBullets` request supports `bulletPreset: "BULLET_CHECKBOX"` for creating checkbox glyphs. However, per Google Apps Script community documentation (Kanshi Tanaike), **the checked/unchecked state of an inserted checkbox is not reliably retrievable through `documents.get()`** — the paragraph's `bullet` object identifies the list/glyph but Google's own reference material does not document a clean boolean "checked" field parallel to what the UI shows (checked items visually get strikethrough text style, which *is* readable via `textStyle.strikethrough`, but that's a proxy, not a first-class field, and collides with any legitimate strikethrough formatting a user might apply for other reasons). Given `docs_structure_parser.py` currently ignores `strikethrough` entirely, whatever checklist scheme gets picked (literal `[ ]`/`[x]` text vs. `BULLET_CHECKBOX` + strikethrough-as-checked-proxy) needs to be validated by writing a real checkbox in the live doc's UI and reading it back with `documents.get()` before trusting the round-trip — don't assume the API shape from docs alone.

**Batch atomicity is a double-edged sword.** Confirmed: batchUpdate validates the full request list before applying anything, and applies atomically — a single invalid request (e.g., a bad index from a checklist edge case) fails the *entire* batch with no partial corruption. Good for safety (no half-applied pushes), bad for iteration speed (any bug in new checklist-request logic will hard-fail every push touching that paragraph, including unrelated changes bundled in the same push) — expect to want small, isolated pushes while iterating on the checklist feature rather than one big batched push.

**Rate limits**: `client.py`'s `_with_backoff` already retries `documents.get`/`batchUpdate` on HTTP 429 with truncated exponential backoff (max 5 retries, up to 64s). This is adequate for a single-user, occasional-sync tool — not a concern at this scale/frequency.

**Idempotency of pull→push cycles**: `push()` always re-reads the live doc fresh via `get_document()` (not the last-known state from `docspan pull`) and diffs against it, so a true no-op edit should produce `requests == []` → `PushResult(status="skipped")`. The risk is *false* non-idempotency: if `MarkdownToParagraphParser` (mistune-based) and `DocsStructureParser` don't produce byte-identical `(style, text, is_list_item)` tuples for visually-identical content — e.g., whitespace trimming differences, or a checklist item that round-trips as `"[ ] Book DJ"` from Docs but is authored as `"- [ ] Book DJ"` and mistune (without the `task_lists` plugin, which is **not enabled** in `markdown_to_paragraph_parser.py:91`) parses the leading `[ ]` as literal text with different whitespace — every pull→push cycle will register a spurious diff on every checklist line and force a delete+reinsert, which is exactly the operation that orphans anchored comments (see §3). This needs a round-trip unit test (Docs JSON → markdown → Docs JSON, assert node-for-node equality) before trusting it against the live doc.

## 2. Live-document, concurrent-editor race conditions

**Two separate race windows exist, and only one of them is addressed by the stated plan:**

1. **Human-timescale race** (minutes to hours): someone edits between Tyler's first `docspan pull` and his eventual `docspan push`. Requirements.md's mitigation — re-pull immediately before push — narrows this window but does not close it, because `docspan pull` and `docspan push` are two separate CLI invocations with a human in between (reviewing the dry-run output). **Given §0, the "review dry-run diff" step that's supposed to catch this is currently not implemented**, so as written today this mitigation is closer to "re-pull and hope" than a real gate.

2. **Machine-timescale race, internal to `push()` itself** (seconds): `GoogleDocsBackend.push()` (`backend.py:49-74`) calls `self._client.get_document(doc_id)` to build `current_nodes`, then later calls `self._client.batch_update(doc_id, requests)`. Between those two calls — the time it takes to run `DocsRequestBuilder().build()`, typically milliseconds but with no upper bound guaranteed — a collaborator could save an edit in the browser. `batch_update()` sends `body={"requests": requests}` with **no `writeControl`** at all. This window exists on *every* push, independent of the CLI-level "pull immediately before push" discipline, and is not mentioned in requirements.md.

**A cheap, concrete guard exists and is currently unused**: the Docs API's `batchUpdate` supports `writeControl: {"requiredRevisionId": "<revisionId>"}`. If the document's actual revision no longer matches, the API returns `400 Bad Request` and applies nothing (confirmed via Google's own developer docs: *"if this is not the latest revision of the document, the request will not be processed and will return a 400 bad request error"* — Google's own guidance explicitly recommends this even for documents you don't expect to be edited concurrently: *"it's important to program defensively... make sure the document state remains consistent"*). `GoogleDocsBackend` already has `get_remote_version()` returning `doc["revisionId"]`, and `push()` already fetches the full `doc` dict via `get_document()` right before diffing — so `doc["revisionId"]` is sitting right there, unused. Wiring `writeControl.requiredRevisionId` into `client.batch_update()` closes race window #2 completely (it becomes a hard API-level guarantee instead of a hopeful narrow-the-window measure) for a very small code change, and is a much stronger and cheaper guard than "re-pull and eyeball a diff" for window #1 too — it fails loudly and safely instead of silently overwriting.

**Realism check on "is re-pull + dry-run sufficient for a Small appetite personal tool"**: re-pull narrows window #1 to roughly the time it takes Tyler to read the diff and decide to push (potentially minutes if reviewing carefully) — not zero, but proportionate to the actual collaboration cadence (occasional comments, not real-time co-editing). It is *not* sufficient on its own given §0 (no diff currently rendered). Recommended minimum bar for Small appetite: (a) implement a real dry-run that shows the request list in human-readable form (not just a stub), and (b) add `requiredRevisionId` to `batch_update`. (a) gives Tyler visibility; (b) gives a hard backstop if he pushes anyway and something changed underneath him in the last few seconds. Both are small, targeted changes consistent with a 1-2 day appetite — no need for locking, presence indicators, or CRDT-style merge for a personal tool with occasional async collaborators.

## 3. Stack-specific risks (google-api-python-client + merge3 three-way merge) applied to checklists

**merge3 operates on markdown lines, not Docs structure — checklist toggles are exactly the kind of edit three-way merge handles worst.** `core/merge.py` calls `Merge3(base_lines, ours_lines, theirs_lines).merge_lines(...)` — pure line-based diffing on the **markdown representation**, invoked in `orchestrator._merge_pull()` only when both local and remote changed since last sync. A checklist toggle (`- [ ] Book DJ` → `- [x] Book DJ`) is a single-line, single-character-class change. If Tyler edits the same checklist line locally (e.g., rewording the task) *and* a collaborator checks it off in the live doc in the same sync interval, merge3 will see two different edits to the same line and — depending on exact overlap — either silently pick one side or emit a conflict-marker block wrapping the entire line, which for a checklist item degrades to visible garbage (`<<<<<<< ours\n- [ ] Book DJ (reworded)\n=======\n- [x] Book DJ\n>>>>>>> theirs\n`) that must be hand-resolved via `docspan conflicts resolve` before the next push can proceed cleanly. This is expected/acceptable per ADR-001, but worth calling out explicitly: **checklist state and checklist text are conflated onto one line**, so any edit to a task's wording collides with any check/uncheck of the same task in three-way merge, even though semantically they're independent changes (text vs. status) that a smarter merge could reconcile separately. Out of appetite to fix (would need structured, not line-based, merge) — but Tyler should expect this specific collision pattern and know `docspan conflicts resolve` is the escape hatch.

**google-api-python-client**: no version pin issue found; `client.py` uses the standard `build('docs', 'v1', ...)` / `build('drive', 'v1', ...)` discovery pattern with reasonable retry/backoff already in place (§1). No new risk specific to checklists here beyond what's covered in §1 (BULLET_CHECKBOX read-back gap) and §0 (writeControl).

**Comment anchoring is not actively managed at all — and checklist toggles are a direct new source of paragraph rewrites that will trigger it.** Grep across the entire `google_docs` backend found zero references to the Drive/Docs comments API (`comments()`, `replies()`, etc.) — confirmed no code reads, tracks, or reasons about comment anchors. Comment preservation today is **purely incidental**: it survives only when `DocsRequestBuilder`'s SequenceMatcher opcode for a paragraph is `"equal"` (text and style unchanged). The moment a paragraph's `(style, text, is_list_item)` tuple differs at all — which is exactly what happens when a checkbox is toggled if checklist state is encoded as part of the paragraph text (literal `[x]`/`[ ]`) — the opcode becomes `"replace"`, and `_make_delete_requests` + `_make_insert_requests` fully **delete and recreate** that paragraph (`docs_request_builder.py:72-80`). Google Docs orphans (or drops) comments anchored to deleted text ranges. **Concretely: if Tyler's workflow ever checks off a task on the same paragraph Nora's two open comments are anchored to, pushing that change will very likely destroy her comments** — this is the README's documented limitation ("comments on edited paragraphs are lost on push") intersecting directly with the new checklist feature, on the literal document in scope. This needs explicit verification against the real doc (create a throwaway comment on a checklist line in a scratch doc, toggle the box, push, check if the comment survives) before Tyler trusts the workflow near Nora's actual comments — do not assume either outcome.

## 4. Deadline-driven design: what to design against, and the fallback

**The deadline is real and immovable — the wedding happens 7/29 regardless of tool readiness.** Given the two structural gaps above (§0: dry-run is a stub; §3: checklist toggles risk destroying live comments), the highest-priority design question isn't "how good can the fix be" but "how do we guarantee the fallback is always available and cheap to reach for."

**Recommended kill-switch design, cheap to build into the Small-appetite scope:**
- **Never let a checklist-toggle edit share a push with anything else.** If Tyler is checking off tasks near Nora's/Bekah's open comments, do that push in isolation (small, reviewable diff) rather than bundled with schedule/housing edits — reduces blast radius if something goes wrong and makes the (currently-missing, needs building) dry-run output small enough to actually read.
- **Treat any paragraph with a known open comment as off-limits for automated push** until comment-anchor survival is empirically verified (§3). Concretely: before pushing, Tyler (or a Claude-assisted check) should note which paragraphs currently have Nora's/Bekah's comments and manually confirm the diff doesn't touch those exact lines — this is a manual checklist, not code, and can be done today with zero new code.
- **The fallback is already trivially available and should stay that way**: the live Google Doc is the source of truth regardless of docspan's state. Because docspan is additive (pull reads, push is explicit and gated behind CLI invocation + human review), **doing nothing** — i.e., Tyler and collaborators just keep editing the Doc directly in the browser, as they do today — remains available at every point with zero migration cost, as long as Tyler doesn't leave the local markdown file as the only place a change lives (i.e., always push promptly, or don't rely on local edits surviving if the tool gets abandoned mid-way). The `.orig` backup and content-addressed `.markgate-base` store (`core/orchestrator.py`) give rollback for *local* mistakes, but note they do **not** protect the live doc from a bad push — that protection is only as strong as dry-run + revisionId guard (§0, §2), which currently need to be built.
- **Go/no-go gate for using the checklist feature near the wedding**: don't gate the whole workflow (pull → Claude summarize → edit) on the checklist fix — that part is safe today (no live-doc writes happen until push) and delivers value (task/owner/due-date summaries) even if checklist push support is dropped. Gate specifically the *push* of checklist-state changes: if by go-time the checklist round-trip isn't verified end-to-end against a scratch copy of the real doc (including the comment-anchor check above), Tyler should keep checking off boxes by hand in the browser (a 2-second action) and use docspan purely for pull + summarize, skipping push for checklist edits. This keeps the hard deadline from forcing a rushed, unverified push against the one document the whole wedding party depends on.

## Summary of concrete, actionable findings for planning phase

1. **`docspan push --dry-run` is a no-op stub** (`cli/main.py:107-111`) — must be implemented for real before the workflow's stated risk control is true. No `diff` command exists as an alternative.
2. **No `writeControl.requiredRevisionId` guard on `batch_update()`** — cheap, standard Docs API mechanism to hard-fail a push if the doc changed underneath it; `revisionId` is already being fetched in `push()` and just needs to be threaded through.
3. **Index-shift handling in `docs_request_builder.py` is already correct** (write-backwards, descending-startIndex sort over requests computed from one unmutated document read) — preserve this invariant when adding checklist requests; don't introduce a second `batch_update()` call or stale indices.
4. **`BULLET_CHECKBOX` checked-state read-back via `documents.get()` is not well-documented/may not be reliable** — verify empirically against a real doc before picking an implementation approach (literal `[x]`/`[ ]` text vs. native checkbox glyph + strikethrough proxy).
5. **Checklist toggles are a direct trigger for the existing "comments lost on edited paragraphs" limitation** — any checklist-state encoding that changes paragraph text will delete+reinsert that paragraph, which is exactly what orphans Nora's/Bekah's anchored comments. Verify survival empirically; keep checklist pushes isolated and away from commented paragraphs until verified.
6. **merge3's line-based three-way merge conflates checklist text and checked-state on one line** — expect conflict markers when task wording and check-state are edited concurrently on the same line; `docspan conflicts resolve` is the existing escape hatch, no fix needed, just expect it.
7. **Fallback (manual editing in the live Doc) is already always available and cheap** — the design should explicitly gate *pushing checklist changes* (not the whole workflow) on empirical verification, so the hard deadline never forces an unverified write to the one shared source of truth.
